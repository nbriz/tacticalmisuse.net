how to tactically misuse

REST APIs

since the dawn of the Web (1.0) making connections between individual websites has been a core aspect of the platforms DNA. central to this dynamic is the Web's <a> element which is used to create links, clickable pieces of text which can point to any other resource on the Web, enabling users to quickly jump from site to site leaving an associative trail (Vannevar Bush) of connections in ur browser's history. it's even possible to create a website which is itself an assemblage of resources pulled from every corner of the Web; the HTML <img> element is used to embed images into a website/app, often these are stored on the same server as the site itself, but they can just as easily embed any image stored on any server by simply pointing to its URL, the same is true for <video>, <audio> && even <iframe> which let u embed entire websites inside of another.

but there are limits, while u can link to any site from any site && embed media, if u wanted to pull other arbitrary data from another server (say the tweets of a particular user or weather data from the NWS, etc) this required complex work-arounds. but then, during the era of Web 2.0 && the initial rise of social media, it became common practice for developers to create REST APIs alongside any site/app containing user generated content (+/or other useful/interesting data). these APIs give other developers access to data && services stored on other servers, if u've ever seen an app (like Uber for example) have Google maps built right into it, that was possible using Google's Maps API, or if u've ever seen an app which let's u schedule ur tweets in some automated way, that was made possible using Twitter's developer API. these APIs have allowed for a much more interconnected (&& perhaps interdependent, for better/worse) Web by design. but despite their intended purpose, these REST APIs can also be misused towards our tactical aims. we can (&& i have) write code which let's us use YouTube's own API to post videos to their platform which undermine their copyright content-id system for example, or we can use the Google Drive API to store encrypted data on their servers, in this way preventing Google from reading + analyzing all our docs && selling the subsequent insights to their customers, allowing us to leverage their "free" cloud storage w/out paying with our privacy.

i'm still in the process of documenting these techniques, but in the meantime u can checkout my 2018 talk an Artist's Guide To Tactical Mis-Use at the FRANK conference in Gainsville, FL (i discuss REST APIs at the ~11:35 marker), +/or my 2017 talk Their Tools, Ur Rules: The Tactical Mis-Use of Online Platforms; at the Radical Networks conference in New York City, NY (i discuss REST APIs at the ~38:48 marker).
